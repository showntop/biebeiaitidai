# 《别让 AI 替代你》当前状态与后续路线图

更新日期：2026-07-17

## 1. 当前结论

项目已经从“核心规则验证”进入“可玩 Demo 体验收口”阶段。现在最重要的不是继续零散微调某个按钮或某个像素，而是把视觉体系、表现层代码、反馈手感、新手前 30 秒这几件事系统化。

一句话判断：

- 核心规则层是稳的。
- Cocos 表现层已经可玩，但 `GameRunner.ts` 过重。
- UI 美术方向已经从早期 Graphics 代码绘制，转为“资产优先，代码辅助”。
- 当前已完成一轮输入、反馈、结算和性能的系统收口，下一阶段应转向真实玩家数据与内容迭代。

## 2. 当前实现状态

### 2.1 已经比较稳的部分

- `assets/scripts/core/` 是纯 TypeScript 规则层，保持零 Cocos 依赖。
- 事件总线、认可度、传送带、道具、AI 状态、关卡、结算等核心系统已经完成。
- `npm test` 已覆盖核心规则与集成验收，当前基线为 94 个测试。
- Cocos 层已经能跑完整主流程：入口页、主界面、长按拖动道具、任务卡结算、认可度反馈、结算页。
- 本轮已完成统一拖拽生命周期、真实 Perfect 判定与奖励事件、有效目标预判、阶段节奏提示、结果分支战报、性能缓存和新手提示，并通过 `typecheck`、94 个单测、`web-mobile` 构建与 390×844 截图验收。

### 2.2 正在形成但还需要收口的部分

- 主界面视觉已经转向暖纸质、友好蓝、手绘描边、资产化 UI。
- 任务卡、按钮、显示器、桌面元素已经做过多轮截图验收。
- 道具交互已经从“点击按钮”改为“长按进入操作区、拖动道具、松手投出”。
- `FxLayer.ts` 已开始承载卡片命中、认可度变化、结算反馈等爽感动效；`ApprovalGaugeView` 负责轻量短战报，不再把事件信息丢掉。

### 2.3 当前主要技术债

- `assets/scripts/cocos/GameRunner.ts` 职责过多：布局、节点创建、状态刷新、交互绑定、页面切换和部分表现逻辑混在一起。
- `assets/scripts/cocos/FxLayer.ts` 后续会继续增长，需要明确只负责事件驱动的瞬时反馈，不应该变成第二个界面总控。
- 文档中大部分早期描述已经收口，但后续新增功能仍必须按运行时 QA 门槛更新记录。
- QA 截图、Playwright 临时文件、Serena 临时目录已进入忽略策略；当前仅保留 Cocos 编辑器本地配置脏文件不提交。

## 3. 当前设计体系判断

### 3.1 正确方向

当前项目应该坚持：

- 资产优先：主容器、按钮、卡片、胶囊、面板尽量由 Sprite 或九宫格资产提供质感。
- 代码辅助：代码负责布局、状态、动画、动态填充、命中特效。
- Graphics 只做动态层：进度条填充、扫光、选中框、拖拽轨迹、闪烁反馈。
- 中心玩法优先：显示器和任务流是第一视觉焦点，办公桌是气氛支撑，不能抢主操作区权重。

### 3.2 统一视觉语言

后续 UI 应统一到这套语言：

- 背景：暖纸质米色。
- 强调色：友好蓝为主，红/绿/黄仅用于状态反馈。
- 形状：大圆角、厚度按钮、轻柔投影、手绘描边。
- 动效：少而准，集中用于按下、命中、危险、奖励、教学完成。
- 字体：标题偏圆润厚重，正文清晰，数字信息要强层级。

## 4. 后续工作路线

### 阶段 A：收口仓库和文档（已基本完成）

目标：让项目状态干净，后续不再被历史 QA 文件和旧文档误导。

任务：

- 已提交处理反馈动效、道具拖拽反馈、新手提示、结算页布局收口和事件短战报。
- 已忽略本地 QA 截图、Playwright 临时目录、Serena 临时目录。
- 已建立 Cocos 运行时 QA 门槛，后续表现层改动必须按该门槛验收。
- 后续若继续沉淀文档，重点应放在“视觉 token 与组件边界”而不是流水账。

验收：

- `git status --short` 不再显示大量截图和工具临时文件。
- `npm run typecheck` 通过。
- `npm test` 通过。
- Cocos 表现层改动必须额外通过 web-mobile 真机比例截图，不允许只用 `tsc` / 单测判断。

### 阶段 A+：Cocos 运行时 QA 门槛

这条规则来自一次真实故障：`tsc` 和 Vitest 都通过，但 Cocos 运行时 UI 没有正常显示。因此后续只要改到 `assets/scripts/cocos/`、场景、资源 `.meta`、UI 资产引用，就必须执行运行时验收。

必做验收：

1. `npm run typecheck`
2. `npm test`
3. Cocos Creator 构建 `web-mobile`
4. 用 390 × 844 视口打开 `build/web-mobile`
5. 截图确认入口页出现
6. 点击进入游戏，截图确认主界面出现
7. 浏览器 console 不应出现 error/warn

注意：

- `web-desktop` 在窄视口下可能仍保留 1280 宽画布，390 截图会只截到画布左侧，看起来像空白；这不是可靠移动端验收。
- 新增或移动 Cocos 脚本文件风险高于普通 TypeScript 重构，因为 Cocos 还涉及脚本注册、`.meta`、场景绑定和构建产物。
- 在没有真实截图验证前，不要提交会影响 Cocos 初始化链的重构。

### 阶段 B：Cocos 表现层模块化（进行中，主要高风险区域已拆）

目标：降低后续 UI 精修成本。

建议拆分：

- `StartPageView`：入口页卡片、状态药丸、三步玩法、主按钮。
- `GameplayHudView`：标题、关卡胶囊、倒计时、顶部状态。
- `MonitorView`：显示器、任务队列、传送带、卡片容器。
- `WorkstationView`：桌子、AI 角色、花盆、杯子、日历、键盘等气氛物件。
- `ApprovalPanelView`：认可度面板、分段条、危险状态、事件提示。
- `PropDockView`：道具按钮、禁用态、长按操作区、拖拽提示。
- `ResultDialogView`：结算弹窗、失败/成功标题、评价格、按钮。

原则：

- `GameRunner.ts` 只做组装和状态分发。
- 组件自己维护节点结构和刷新接口。
- 视觉 token 从 `UiTokens.ts` 取，不在各处散落魔法数字。
- 在 Cocos 初始化链稳定前，优先做 `GameRunner.ts` 内部整理，不贸然新增脚本文件。

### 阶段 C：反馈爽感与可玩性（本轮完成）

目标：让玩家每次操作都有“我影响了局面”的感觉。

已完成：

- 任务卡命中反馈：弹性、盖章、爆点、Perfect 金色反馈。
- 任务结算反馈：扫描、飞向认可度、数字滚动、正负颜色统一。
- 倒计时 10 秒内红色脉冲；中盘/危机阶段切换有局部状态提示。
- AI 状态反馈、Boss 局部预警、反杀窗口、复活与冻结反馈。
- Perfect 随机奖励明确展示，结算记录有效命中、Perfect 与失误次数。
- 移除大面积红/黄闪屏，以局部横幅、边缘刻度、目标圈和轻震替代。

### 阶段 D：新手前 30 秒（本轮完成）

目标：让玩家第一次进入就能理解“长按、拖动、松手丢出去”。

已完成：

- 第一次出现可用道具时，弹出米白纸质指引卡。
- 长按成功后由滑轨自身承担教学，提示左右选目标和松手投出。
- 无效目标在松手前变灰并提示换目标。
- 第一次稳定对准时解释金色准星与 Perfect 奖励。
- 第一次命中后结束强引导并永久记录，后续不常驻占屏。

### 阶段 E：内容与商业化外围

目标：Demo 稳定后再扩内容，不要太早堆关卡。

任务：

- 先做 10–20 个验证关卡，而不是直接做 60–80 个正式关卡。
- 按“3 难 + 1 爽 + 1 节点”的节奏验证留存。
- 再做日报/战报分享图、排行榜、复活/广告点。
- 后期再接入每日挑战与社交裂变。

## 5. 后续产品化执行顺序

当前最推荐的顺序：

1. 接入首关关键埋点：开始、首次长按、首次命中、首次 Perfect、失败原因、复活、通关路径。
2. 用 20–50 名陌生玩家验证前 30 秒上手率，优先看“是否知道长按”和“是否理解认可度方向”。
3. 根据数据调 `perfectLockSec`、道具 CD、首关生成速度和危险阈值，不凭单次截图调数值。
4. 补一套体积受控的正式音效资产：按下、投出、命中、Perfect、危险、胜负。
5. 验证 10–20 关内容节奏后，再考虑分享图、排行榜和商业化入口。

暂时不建议：

- 继续无序微调单个 UI 元素。
- 立刻扩大量关卡。
- 立刻做广告、分享、排行榜。
- 继续把所有页面和动效都堆在 `GameRunner.ts`。

## 6. 爆款准备路线（下一阶段执行基线）

“爆款”不可由功能数量保证，但可以把产品做成具备被验证、被传播和快速迭代的状态。当前版本已有 20 关与完整主循环，下一阶段不再以继续堆 UI 为主，而是按下面顺序推进。

### P0：冻结可测基线（1～2 天）

目标：建立一个任何后续改动都能对比的稳定版本。

- 收口并提交当前交互、UI、反馈和结算改动。
- 把 `web-mobile 390×844`、微信包体、94 项测试作为固定回归门槛。
- 增加首局可复现 seed 与 QA 快捷入口，方便直接进入拖拽、Perfect、危机和三种结算状态。
- 保留现有 Cocos 层本地配置，不把编辑器生成噪声混入产品提交。

验收：同一 seed 的关键任务序列与结算结果可重复；入口、主界面、拖动、胜负三种结算均可一键截图。

### P1：首局漏斗与失败原因埋点（2～3 天，最高优先级）

目标：先知道玩家到底卡在哪里，再调手感和数值。

首批事件：

- `session_start / level_start / level_end`
- `tutorial_shown / prop_hold_started / first_drag / first_release`
- `first_valid_hit / first_perfect / invalid_target / gesture_cancel`
- `approval_zone_changed / boss_warning / revive_used`
- `result_type / fail_reason / retry / next_level / return_home`

每局同时记录：设备档位、FPS 分位、首击耗时、有效命中率、Perfect率、取消率、道具使用分布、认可度峰值和最终结果。

第一轮内部目标值（是产品假设，不是行业基准）：

- 进入后 10 秒内开始游戏 ≥ 85%。
- 30 秒内完成第一次有效命中 ≥ 75%。
- 手势意外取消率 ≤ 5%。
- 首关完成率 ≥ 65%。
- Perfect率保持在 10%～25%，既能感知又保留练习空间。

### P2：核心手感闭环（3～5 天）

目标：让“按住—拖动—锁定—投出—命中”每一步都有清晰、克制且递进的反馈。

- 接入正式音效：按下、拾起、滑轨移动、锁定、投出、命中、Perfect、危险、Boss、胜负。
- 增加可关闭的轻震反馈，并对触发频率限流。
- 让四种道具有重量差：起手缩放、跟手滞后、飞行弧线、落点音色和命中特效不同。
- 完善 Miss、无效目标与取消手势：明确但不惩罚，不让玩家误以为游戏失灵。
- 对低端机保持对象池和绘制缓存，强特效只用于 Perfect、反杀、Boss 与结算。

验收：陌生玩家不看说明也能描述四步操作；连续拖动无明显掉帧；静音状态仍能仅凭画面读懂结果。

### P3：前 10 分钟内容与留存节奏（5～7 天）

目标：让现有 20 关从“配置数量”变成有记忆点的内容体验。

- 关卡 1～3：只教核心动作，不叠加复杂变量。
- 关卡 4～6：引入道具取舍与第一轮爽关。
- 关卡 7～10：引入拍马屁、Boss Tell 与第一次明确反杀目标。
- 关卡 11～20：加入固定 seed 挑战、限道具目标、Perfect目标和高权重卡组合。
- 保持“3 难 + 1 爽 + 1 检查点”的锯齿节奏；每 3～5 关出现一个可讲述的新梗或新规则。
- 结算页增加明确的下一关钩子，而不是只显示“下一关”。

验收：玩家首次会话能连续玩 3 局以上；失败后重试率和通关后下一关点击率均可从埋点直接看到。

### P4：AI 人格与高光时刻（3～5 天）

目标：把“办公室题材”变成玩家愿意记住和转述的角色关系。

- AI 增加 4～6 个高辨识状态：装忙、得意、慌张、捂脸、死机、被老板叫走。
- 事件播报从系统提示升级为短吐槽，但每次只出现一句，避免信息噪声。
- 建立高光规则：三连击、连续 Perfect、Boss 极限拦截、1 秒反杀、危险区翻盘。
- 结算根据高光生成称号和一句梗文案，为后续分享卡直接提供素材。

验收：一局至少有 1 个可截图高光；AI 状态变化能被玩家准确描述，而不是只看到数值变化。

### P5：传播闭环（3～5 天）

目标：让传播内容本身有趣，而不是把“分享”做成按钮。

- 先做结算战报图：反杀类型、存活类型、极限翻盘类型各一套。
- 分享指标全部采用“越大越好”的正向表达：AI劝退次数、最高连击、Perfect数、存活天数和职场段位。
- 增加固定 seed 的每日挑战/好友挑战，使分享链接对应同一局面，结果可比较。
- 分享必须自愿，奖励只做锦上添花，不阻断主流程。
- 排行榜和群挑战等到战报分享被真实使用后再接，不提前做空壳社交系统。

小游戏分包应按场景和资源依赖规划，独立分包不能依赖主包或其他分包；正式音频、后续关卡和分享资源应优先进入分包，保持首包启动轻量。

### P6：留存与轻量成长（3～5 天）

目标：给玩家第二天回来一个理由，但不把小游戏变成数值养成表格。

- 每日挑战：同 seed、同道具、同任务流，比结果与高光。
- 轻收藏：AI表情、工位摆件、纸团皮肤、战报边框，不提升战斗数值。
- 成就：第一次反杀、三连 Perfect、Boss 极限拦截、无伤生存。
- 段位展示服务于战报和好友比较，不强制每日打卡。

灰度目标值（需用真实数据校准）：D1 ≥ 25%、D3 ≥ 12%；若未达到，不先加更多成长系统，而是回查首局漏斗和前三关节奏。

### P7：商业化与正式发布（最后接入）

目标：在核心体验成立后变现，不让广告破坏首局。

- 激励视频只放在失败复活、结算奖励翻倍或额外挑战次数等玩家主动选择点。
- 首局不弹插屏；广告频次必须与留存、重试率一起做灰度对照。
- 微信构建继续使用引擎插件与分包；低端安卓、iPhone 刘海屏、弱网和后台切回做专项测试。
- 上线前完成隐私、分享文案、广告奖励、敏感职场用语和审核素材自查。
- 建立崩溃、启动耗时、资源加载失败和卡死监控。

### Go / No-Go 节点

在扩大内容、接排行榜或增加投放前，至少满足：

1. 20～50 名陌生玩家完成无讲解试玩。
2. 首击、取消、首关完成和失败原因可量化。
3. 大多数玩家能复述核心操作和两种胜利方式。
4. 至少一个高光事件能自然促使玩家截图或主动转述。
5. 低端机 60 秒完整对局没有持续卡顿或阻断性错误。

未满足时，继续修首局、手感和前三关；满足后再进入传播、留存和商业化扩张。
