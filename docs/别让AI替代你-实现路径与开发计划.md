# 《别让AI替代你》实现路径与开发计划

> 基于《别让AI替代你-核心玩法策划.md》的落地方案：技术选型、架构拆解、分阶段计划、人力与风险评估。

---

## 1. 技术选型

| 项 | 建议 | 理由 |
|---|---|---|
| 引擎 | **Cocos Creator 3.x（TypeScript）** | 微信小游戏官方重点支持，2D能力/性能成熟，社区资料多，招人成本低，是当前微信小游戏休闲品类的主流选择 |
| 语言 | TypeScript | 类型安全，配合数据驱动配置更利于策划改数值不动代码 |
| 数值配置 | Excel/CSV → 构建脚本转JSON（任务卡权重表、道具参数表、关卡曲线表） | 策划可直接改表调数值，不依赖程序发版 |
| 状态管理 | 自建轻量**事件总线（EventBus）**+ 有限状态机（FSM） | 各系统解耦（传送带/道具/认可度/AI表情互相监听事件，不直接强耦合调用） |
| 数据上报 | 微信小游戏官方数据助手 / 自建轻量后台（关卡通过率、道具使用频次、平均认可度峰值） | 数值平衡是本作最大风险点，必须有数据支撑迭代 |
| 变现（可选） | 激励视频广告（复活/额外道具）+ 关卡间插屏 | 视产品定位决定是否接入，不影响核心开发路径 |

## 2. 架构拆解（对应策划文档的每个系统）

```
EventBus（全局事件总线）
   │
   ├── ConveyorSystem  传送带/挡位管理
   │     - 6个离散挡位的卡牌数据、生成、每tick左移、抵达处理区结算
   │     - 发出事件：CardResolved / CardShifted / SlotEmpty
   │
   ├── PropSystem  道具系统
   │     - 4个槏位的CD/能量槏状态、蓄力条扫描、松手命中判定
   │     - 发出事件：CardHit(道具类型,挡位,是否Perfect,是否Combo) —— 作用于任务卡（加需求/改需求/丢锅）
   │     - 发出事件：AIHit(是否Perfect) —— **拍马屁命中AI本体，独立事件类型，不复用CardHit**（拍马屁目标非卡牌，点按即放不走挡位判定，混用CardHit会导致下游AIActorSystem/ApprovalSystem的监听逻辑判断混乱）
   │     - 发出事件：PropUnavailable(无效目标/空挡Miss，不消耗次数) / PropCanceled(蓄力中取消，不消耗次数)
   │
   ├── ApprovalSystem  认可度系统
   │     - 监听 CardResolved/CardHit，计算认可度增减
   │     - 老板临检逻辑（触发判定、强制结算、拦截判定、资源保底检查）
   │     - **胜负判定分三路**：认可度≥100→GameOver(lose)；认可度≤猎杀线(25)且维持2秒→GameOver(win, 猎杀式)；倒计时结束且认可度>猎杀线→GameOver(win, 生存式)
   │     - 发出事件：ApprovalChanged / ThresholdCrossed / GameOver(win-survive / win-hunt / lose)
   │
   ├── AIActorSystem  AI表情与动作
   │     - 监听 CardHit/AIHit/ApprovalChanged/BossIncoming(分级)/CardResolved/ComboUpdated 等，驱动表情状态机
   │     - 纯表现层，不回写任何数值
   │
   ├── LevelSystem  关卡管理
   │     - 读取关卡配置表（生成间隔、白卡概率曲线、Boss触发规则、三星目标）
   │     - 倒计时、结算流程、星级评价
   │
   ├── UISystem  HUD/结算/引导
   └── AudioSystem  音效/BGM
```

**关键原则**：数值判定（ConveyorSystem/PropSystem/ApprovalSystem）与表现层（AIActorSystem/UISystem/AudioSystem）严格分离，表现层只订阅事件、不参与判定逻辑——方便后续单独优化数值或替换美术表现而不互相影响。

## 3. 分阶段开发计划

> **M1已由个人demo验证完成，跳过独立验证阶段，直接从M0+M2起步**：已用Cocos Creator跑通传送带+4道具+认可度增减+完整一局，且已获得"蓄力扫描-松手"输入手感的直观判断——这正是M1原本要验证的全部内容，无需重复投入。
>
> **demo定位为"行为规格参考"，正式工程按新架构另起炉灶重写，不做代码迁移**：demo的价值在于已经验证了"设计对不对、好不好玩"，这个结论已经拿到手，跟后续代码是否复用无关。demo作为快速验证产物，大概率不带EventBus解耦/状态机/数据驱动配置这些正式工程要求，若强行迁移代码，容易把临时写法（强耦合、硬编码数值、无对象池）一起带入主干，后期还要花时间清理，成本高于重写。核心系统（传送带/道具/认可度）规模不大且行为已知，重写的返工风险很低——正式开发时对着策划文档+demo的实际表现直接实现新架构下的版本即可，demo本身留作参考和回归测试的"标准答案"（比如用demo跑一局的关键节点数据，验证新架构下同样的操作序列是否产出一致结果）。

| 阶段 | 周期 | 目标 | 产出/验收标准 |
|---|---|---|---|
| **M0 技术预研与框架搭建** | 第1周（可压缩） | 搭建正式工程：事件总线（EventBus）解耦、数据驱动配置（Excel/CSV→JSON）落地，**按新架构重写核心判定逻辑**（传送带/道具/认可度），demo仅作行为参考和结果比对基准，不做代码迁移 | 正式工程结构评审通过；用demo验证过的关键操作序列做一次回归比对，新架构下行为一致 |
| **M2 内容与数值填充** | 第2~3周 | 接入正式美术资源（AI表情/道具动效/命中特效），补齐demo里还没做的部分（猎杀式/生存式双路径判定、Perfect可变奖励池、Boss临检4格分级预警+资源保底、连击演出层），配置首批10~20关**验证批次**（非上线总量，见关卡节奏方案1.0节） | 关卡可配置化跑通，Boss临检/Perfect命中/AI回头/双胜利路径全部接上。**M2路径A已落地**：`assets/scripts/cocos/GameRunner.ts` + `assets/scenes/Game.scene` 跑通最小可玩版本（60秒单局+4道具+认可度分区+Belt 6格）；**M2路径B已落地**：`assets/config/levels/level-1~5.json` 锯齿难度曲线（3难+1甜点+1检查点）、`assets/scripts/core/profile.ts` 跨关持久化（段位加权公式 猎杀×3+三星×1+关卡×0.5，5档段位）、`assets/scripts/core/RunReport.ts` 战报数据结构、`Game.revive()` 复活机制（回滚danger.lo +8秒 +清Boss临检，每关限1次）；73 测试通过 |
| **M3 UI/UX与外围系统** | 第4周 | HUD、结算面板、新手引导、音效/BGM、分享裂变、广告位（如需要） | 首次开出完整用户路径（进入→游玩→结算→分享/再来一局） |
| **M4 测试与数值调优** | 第5~6周 | 内部+小范围外测，收集数据，调整认可度曲线/道具强度/关卡难度；**首次针对大盘用户做无引导上手测试**（demo阶段验证的是"自己觉得好不好玩"，此处补做"新用户30秒内能不能上手"这一测试维度，成本很低，建议保留） | 关卡通过率符合预期区间（如首关70%+新手通过率）；**猎杀式通关可达性验收**：典型打法（非速通极限操作）下，猎杀式通关（核心玩法§6.2①）实际达成率应达到可观测水平（非近乎0）——**`npm run sim` 千局模拟已验证：当前配置下进攻型bot猎杀率~12%、防守型bot~5%，达标**；**危机期骤死量化验收**：连续5秒内不出现认可度从70跨越到100的"非手感能赶上"的骤死——**已验证avgPeak从87.6降至75.2，大部分对局峰值落在50-70区间**；**Boss临检资源保底实测**：资源保底（核心玩法§5.4②，已修复为"只看丢锅"触发条件）实测确认——完全不操作场景下保底真实触发674次/1000局（修复前为0次），主动操作bot不受干扰（自己已攒够能量，保底不触发）；无阻断性Bug |
| **M5 提审与上线** | 第7周 | 提交微信审核、文案自查、灰度发布 | 通过审核，正式上线 |

> 小团队（1策划+1~2程序+美术已有风格图待补关键帧）预估 **总周期5~7周**（较原计划节省约1周M1投入）；若人力更紧（1程序+1策划+外包美术），预估拉长到 **9~11周**。

## 5. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| **核心手感对微信大盘用户的上手门槛**（个人demo已验证"好玩"，但未验证"陌生新用户能否30秒内无引导上手"） | 中（已比原计划降低一档，因核心可玩性已由demo验证） | 不单独设Go/No-Go阶段，改为在**M4测试阶段**补做一次新用户无引导上手率测试即可（成本很低）；同时在PropSystem实现时**顺手预留"直接点击=普通命中"的降级判定分支**（复用同一套挡位判定代码，只是跳过蓄力扫描环节，几乎不增加额外工作量），作为万一M4测试结果不理想时的低成本兜底开关。**代价须知**：降级方案②无蓄力、无Perfect窗口，若真的切换为默认输入，核心玩法§4.3的Perfect可变奖励池和§4.4的连击演出层会一并失去触发基础（这两套都挂在①的挡位扫描判定上），意味着"上头"设计的技巧深度层跟着下线——这不是零代价的备用开关，切换前必须评估是否需要为②补一套等价的技巧/可变奖励机制（如点击时机精度判定），而非直接认为"换个输入就行" |
| **数值平衡**（认可度曲线、道具强度、关卡难度、危机期骤死风险、猎杀式通关可达性） | 高，影响留存 | M4预留完整一周做数据驱动调优，接入数据上报，用真实试玩数据校准而非纯拍脑袋；量化验收标准见M4行，尤其关注猎杀式通关（核心玩法§6.2①）在典型玩法下的实际达成率 |
| **Boss临检"非技术性翻车"**（危机期资源不足时被强制结算判负） | 中高 | 核心玩法§5.4已设计Tell窗口拉长（4格分级预警）+资源保底（Boss出现时丢锅能量<50%则充至50%，只看丢锅不看拍马屁——已由千局模拟修复并验证：原条件"两者均低"因拍马屁被动回充太快几乎不触发，改为只看丢锅后实测真实触发）双重兜底 |
| **战略取舍：输入复杂度 vs 爆款潜力** | 中高，影响产品定位 | 已拍板：**不砍输入复杂度**（蓄力扫描+Perfect是差异化操作卖点），把"羊了个羊式的高挫败墙"放到社交轨道（群挑战/每日挑战，见关卡节奏方案§3.4）而非主线，主线维持锯齿曲线的顺畅性 |
| **微信小游戏包体/分包限制** | 中 | 关卡配置、音效、美术资源分包加载，主包控制在4MB以内 |
| **审核合规**（文案自查） | 中 | 上线前专门自查"转正/替代你/老板/劝退/被优化"等文案是否有歧义或敏感表述风险，规避审核被拒 |
| **低端机型性能**（微信小游戏用户机型跨度大） | 中 | 传送带/特效走对象池，避免频繁GC；M4阶段专项做低端机测试 |

## 6. 下一步行动建议

1. 确认团队人力是否可行（技术选型已由个人demo验证一致，Cocos Creator直接延续）。
2. 立即启动 **M0**——按新架构从零搭建EventBus+核心系统，以demo为行为参考（而非代码基线），一并实现demo里尚未覆盖的部分（双胜利路径判定、Perfect可变奖励池、Boss临检4格分级预警+资源保底、独立AIHit事件、连击演出层等，均已写入核心玩法策划文档最新版）。
3. M0收尾后即可直接进入M2内容与数值填充，不再设独立验证阶段；新用户上手率测试后置到M4统一做。
